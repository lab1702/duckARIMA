# Specification: SARIMAX in Pure DuckDB SQL, Validated Against statsmodels

**Status:** Draft v3 · **Target:** DuckDB ≥ 1.5.4 ("Variegata" line), forward-compatible with DuckDB 2.0 (September 2026) · statsmodels ≥ 0.14 as reference implementation

## 1. Goal

Implement seasonal ARIMA with exogenous regressors — full SARIMAX estimation, inference, and forecasting — entirely in DuckDB SQL, with no user-defined functions, extensions, or external computation at runtime. The implementation is validated against statsmodels' `SARIMAX` at three tiers: exact agreement of the Kalman-filter loglikelihood at fixed parameters, agreement of fitted parameters (including regression coefficients) to optimizer tolerance, and agreement of forecasts and intervals to matching tolerance.

The central design principle is that the loglikelihood function is the contract. At any fixed parameter vector, the loglikelihood is deterministic arithmetic and must reproduce statsmodels to near machine precision; this is the brutal unit test that validates every layer beneath it. The argmax of that function, by contrast, is only defined to optimizer tolerance — in statsmodels itself as much as here — so parameter-level agreement targets are set accordingly and no attempt is made to replicate L-BFGS-B iterate sequences.

## 2. Non-Goals

The following are explicitly out of scope for v1. Trend terms (`trend='c'`, `'t'`, etc.) — a constant can be supplied by the caller as an exog column of ones, which is mathematically equivalent under the pinned options, so a separate trend code path buys nothing; the fixture generator therefore always passes `trend='n'`. Dynamic regression beyond linear contemporaneous exog: no distributed-lag or transfer-function structures (callers can materialize lagged exog columns themselves), and no `mle_regression=False` mode (regression coefficients in the state vector) — v1 pins statsmodels' default `mle_regression=True`, where the β coefficients are MLE parameters and exog enters as an observation intercept. Automatic order selection (auto-ARIMA style unit-root testing and stepwise search); the harness supports fitting a user-supplied grid of orders and comparing by AIC/BIC, but selection logic is the caller's problem. Bit-for-bit replication of statsmodels' returned numbers, which is ill-posed across BLAS backends and hardware. Missing-value handling in the observation series or the regressors. Exact-diffuse initialization (Durbin–Koopman) is deferred to v2; v1 pins the reference to configurations that avoid it (see §5.3). Performance beyond "practical for series of a few thousand observations."

## 3. Success Criteria

All comparisons are made against golden fixtures generated by a pinned statsmodels version with pinned model options (§8). Tolerances are defined on double-precision values.

| Tier | Quantity | Comparison point | Tolerance |
|------|----------|-----------------|-----------|
| T1 | Loglikelihood ℓ(θ) at fixed θ | statsmodels `loglike(θ)` at 25+ probe points per fixture, including the fitted θ̂ | abs ≤ 1e-8 or rel ≤ 1e-10, whichever is looser |
| T1 | Per-timestep filter outputs (vₜ, Fₜ) | statsmodels `FilterResults` | rel ≤ 1e-9 |
| T2 | Fitted parameters θ̂ | statsmodels `fit()` result | abs ≤ 1e-6 per parameter (transformed and untransformed space both checked) |
| T2 | σ̂² and maximized ℓ(θ̂) | same | rel ≤ 1e-8 for ℓ; 1e-6 for σ̂² |
| T3 | Point forecasts, h = 1…36 | `get_forecast` | rel ≤ 1e-6 |
| T3 | Forecast standard errors | same | rel ≤ 1e-5 |
| T3 | Parameter standard errors (numerical Hessian) | statsmodels `bse` | rel ≤ 1e-3 |

The looser tolerance on parameter standard errors reflects that both sides compute them from numerically differentiated Hessians with different step choices; 1e-3 relative agreement is evidence of correctness, not a compromise. The "fitted parameters" rows include the regression coefficients β, tested at the same 1e-6 tolerance as the ARMA coefficients — β is unconstrained, so no transform-space caveat applies. All Tier-3 forecast comparisons are conditional on the same future exog values supplied to both implementations; forecast variance treats future exog as known constants, matching statsmodels' convention.

A fixture passes only if every tier passes. The suite passes only if every fixture passes. T1 failures are always implementation bugs and are never waived. A T2 failure with a T1 pass indicates an optimizer issue (ours or a local-optimum discrepancy) and is diagnosed by comparing ℓ at the two candidate optima.

## 4. Architecture

The system is organized as six layers, each a set of SQL files with a documented relational interface. Layers communicate through tables and views with fixed schemas, never through session state.

Layer 0 is relational linear algebra. Layer 1 is series preparation. Layer 2 constructs the state-space system matrices from a parameter vector. Layer 3 is the Kalman filter and loglikelihood. Layer 4 is estimation: starting values, the stationarity-enforcing parameter transform, and a BFGS optimizer. Layer 5 is forecasting, intervals, and the fit-and-report harness.

### 4.0 Model definition and sign conventions

The model, written out so no implementer has to reverse-engineer signs from test failures. Let wₜ = Δᵈ Δ_s^D yₜ and x̃ₜ = Δᵈ Δ_s^D xₜ (Layer 1). Then

φ(L) Φ(Lˢ) (wₜ − x̃ₜ′β) = θ(L) Θ(Lˢ) εₜ,  εₜ ~ iid N(0, σ²),

with φ(L) = 1 − φ₁L − … − φₚLᵖ, Φ(Lˢ) = 1 − Φ₁Lˢ − … − Φ_P L^{Ps} (AR coefficients enter with MINUS signs in the polynomial, positive in the parameter vector), and θ(L) = 1 + θ₁L + … + θ_qL^q, Θ(Lˢ) = 1 + Θ₁Lˢ + … + Θ_Q L^{Qs} (MA coefficients enter with PLUS signs) — statsmodels' convention, opposite in MA sign to some textbooks (Box–Jenkins writes θ with minus). Stationarity constrains the roots of φ·Φ outside the unit circle; invertibility constrains θ·Θ likewise. The reduced-form polynomials φ*(L) = φ(L)Φ(Lˢ) (order p + sP) and θ*(L) = θ(L)Θ(Lˢ) (order q + sQ) fill the Harvey companion form: with k = max(p + sP, q + sQ + 1), the transition T has φ*ᵢ in column 1 (rows i = 1…k, zero-padded) and the superdiagonal identity T[i, i+1] = 1; selection R = (1, θ*₁, …, θ*_{k−1})′ zero-padded; design Z = (1, 0, …, 0). When any sign or layout question arises, the fixture's exported `ssm` matrices are normative and §5.2's 1e-14 agreement test settles it — the equations above exist so the first attempt is right, not to override the fixtures.

### 4.1 Data model conventions

A time series is a table `series(t BIGINT, y DOUBLE)` with t dense and gap-free from 1. Exogenous regressors live in a companion table `exog(t BIGINT, j INT, x DOUBLE)` in long form (j = 1…r indexing the regressors), covering every in-sample t and — for forecasting — every out-of-sample t up to the requested horizon; a wide-form view is derived where convenient. Matrices are represented relationally as `(name VARCHAR, i INT, j INT, v DOUBLE)` with 1-based indices and dense storage of structural zeros for any matrix that participates in elimination or the Lyapunov solve; filter recursion state may instead be carried as `DOUBLE[]` lists (row-major flattening, with `(i-1)*ncol + j` addressing) inside recursive CTEs where the relational form is impractical. Both encodings must round-trip through documented conversion macros `mat_to_list` / `list_to_mat`. Parameter vectors are tables `params(k INT, name VARCHAR, v DOUBLE)` with the canonical ordering: regression coefficients β₁…β_r, then AR coefficients φ₁…φₚ, MA coefficients θ₁…θ_q, seasonal AR Φ₁…Φ_P, seasonal MA Θ₁…Θ_Q, then σ² — matching statsmodels' parameter ordering (exog first under `trend='n'`) so fixtures compare positionally.

A configuration table `sarima_spec(p INT, d INT, q INT, P INT, D INT, Q INT, s INT)` with exactly one row drives every layer; the regressor count r is not part of the spec row but is derived as `max(j)` from the `exog` table (zero rows meaning r = 0), so the pure-SARIMA and SARIMAX paths are configured identically. The public interface follows the duckLM convention: a single file of `CREATE OR REPLACE MACRO` statements loaded once per session with `.read`, exposing table macros that take table and column names as strings — `sarimax_fit(...)` returns a model table the caller materializes with `CREATE TABLE m AS SELECT * FROM ...`, and downstream macros (`sarimax_forecast`, `sarimax_summary`, `sarimax_evaluate`) take the model-table name as a string. duckLM demonstrates that a full IRLS fit — recursive-CTE optimizer, list-lambda linear algebra, in-SQL covariance inversion — packages inside table macros; this project holds itself to the same bar, with internal helper macros (prefixed `_sarimax_`) decomposing the pipeline where a single expression would be unmaintainable. The `sarima_spec` and `exog` conventions above describe the internal dataflow those macros construct, not something the caller assembles by hand.

### 4.2 Determinism requirement

Every aggregation whose result feeds a T1 comparison must have a defined summation order, because SQL `SUM` over a parallel scan is not order-deterministic and floating-point addition is not associative. The mandated pattern is ordered list aggregation followed by a left-to-right fold: `list_sum` is not assumed ordered; use `list_reduce(list(x ORDER BY t), lambda a, b: a + b)` or an equivalent explicitly sequenced construct. All lambdas throughout the codebase use the Python-style `lambda x:` syntax exclusively — the arrow form (`x -> x + 1`) is deprecated with a warning in 1.5 and will be removed in DuckDB 2.0 — and the test harness runs with `SET lambda_syntax = 'DISABLE_SINGLE_ARROW'` so any regression to arrow syntax fails CI rather than warning. DuckDB's fast list kernels (`list_inner_product`/`list_dot_product`) are permitted as an optimization for matrix-vector rows inside the filter *only after* a dedicated determinism test confirms they produce results identical to the explicit sequenced fold on the project's fixtures at all tested thread counts; absent that confirmation, the explicit fold is used on every T1 path. Matrix products inside the filter must fix the k-index summation order ascending. This costs performance and is applied only where bit-stability matters (filter recursion, loglikelihood accumulation, elimination pivoting); bulk feature construction may use plain aggregates. Additionally, `SET threads = 1` is recommended for validation runs and the test harness must pass at any thread count within tolerance.

### 4.3 DuckDB 1.5 feature baseline

Raising the floor to 1.5.4 changes several "recommended" items to load-bearing requirements and adds new levers. Recursive CTE `USING KEY` (stable since 1.3) is now mandatory, not optional: it is the mechanism for both the filter's per-timestep state carry and the batched multi-probe evaluation of §5.4, and no fallback formulation without it is maintained. Scalar configuration flows through `SET VARIABLE` / `getvariable()` in the layered development scripts rather than string-templated literals; in the assembled macro library the same values arrive as macro parameters, so the two forms are one convention (named typed scalars, never text substitution) at two packaging stages. `MERGE INTO` (1.4) is the sanctioned way for the optimizer driver and grid runner to upsert per-iteration and per-model state tables, replacing delete-and-insert churn. `TRY` expressions (1.3) wrap the numeric guard points of §6 so that a poisoned probe point in a batched gradient evaluation yields a NULL loglikelihood for that probe — which the line search treats as rejection — instead of aborting the whole batch.

Two 1.5 engine improvements are exploited passively but should inform design choices: the expanded applicability of hash joins to join conditions that previously fell back to nested-loop plans directly benefits the relational matrix-algebra layer, whose multiply is an equi-join on inner index and whose Kronecker product uses arithmetic join keys — Layer 0's acceptance suite gains an `EXPLAIN ANALYZE` assertion that its multiply and Kronecker templates produce hash-join plans, treated as a performance regression test; and the 1.4–1.5 aggregate-function optimizations improve the bulk (non-T1) aggregation paths for free. During development, the 1.5 CLI's `_` last-result reference and the pager materially improve the debugging loop for inspecting filter traces, though nothing in the deliverables depends on them.

Considered and rejected for this project: the `VARIANT` type (attractive for heterogeneous config/report payloads, but every hot-path value here is a `DOUBLE` and typed tables are strictly better for the fixtures contract), the built-in `GEOMETRY` type and spatial machinery (inapplicable), Vortex/lakehouse formats (fixtures use Parquet, §8), and the experimental PEG parser, which is left at its default (disabled) because the reference environment should carry no experimental toggles. The 2.0 forward-compatibility requirement is concrete: no deprecated arrow lambdas anywhere (enforced per §4.2), no reliance on behavior slated for change in the 2.0 release notes at time of writing, and CI runs the suite against both the pinned 1.5.4 and the latest available release so breakage from the September 2.0 release is detected the week it ships, not when a user hits it.

## 5. Layer Requirements

### 5.0 Layer 0 — Relational linear algebra

Provides, as macros or CTE templates over the `(name,i,j,v)` encoding: matrix multiply (join on inner index, GROUP BY outer indices, ordered summation per §4.2), transpose, addition/scaling, Kronecker product (index arithmetic on a cross join), vec/unvec, identity and zero constructors, and outer products. Provides a general dense linear solver via Gaussian elimination with partial pivoting as a recursive CTE over the augmented matrix: the recursion is over pivot column k, each step selecting the max-|v| pivot row at or below k, swapping, and eliminating. Must handle systems up to 256×256 (sized for the vec-trick Lyapunov solve at state dimension k ≤ 16) and report a failure flag when the selected pivot magnitude falls below 1e-12. Provides matrix inversion via the same elimination against an identity augmentation, and a symmetric-matrix solve used by BFGS and the Hessian inversion.

Acceptance: solver reproduces NumPy `linalg.solve` on 50 random fixtures (conditioned and mildly ill-conditioned) to rel ≤ 1e-10; Kronecker and multiply verified against NumPy on random fixtures.

### 5.1 Layer 1 — Series preparation

Applies non-seasonal differencing d times and seasonal differencing D times (lag s) via window functions, producing `series_diff(t, w)` with t re-densified and the (d + D·s) initial values recorded in a `diff_anchors` table sufficient to exactly invert the differencing later by cumulative summation. Exogenous columns are differenced identically and in lockstep — under `simple_differencing=True` statsmodels differences both endog and exog, and the regression is fit on the differenced scale — producing `exog_diff(t, j, x)` aligned row-for-row with `series_diff`. Layer 1 also validates the regressors: full in-sample coverage with no NULLs, and a rank check on the differenced design matrix (via the Layer 0 elimination pivots on X′X) that fails loudly with the offending column named if the smallest pivot falls below 1e-10 · trace — a constant column, for instance, differences to zero and must be rejected here rather than surfacing as a singular solve three layers later. Also provides lag-matrix construction `lags(t, y, y_l1 … y_lm)` for the Hannan–Rissanen stage, and the classic sample ACF/PACF as diagnostics (not used in estimation).

### 5.2 Layer 2 — State-space construction

Given the spec row and a parameter vector, materializes the Harvey-representation system matrices as relational matrices: transition T (companion form on the expanded polynomial coefficients), selection R, design Z, observation variance H = 0, and state variance Q = σ². With exogenous regressors under `mle_regression=True`, the regression enters not through the state but through a time-varying observation intercept dₜ = xₜ′β — the only time-varying system quantity in the model — materialized as a table `obs_intercept(t, d)` computed from `exog_diff` and the β block of the parameter vector in one grouped join (ordered summation per §4.2, since dₜ feeds T1 arithmetic). All other system matrices remain time-invariant, so the Lyapunov initialization and every structural property of the filter are untouched by exog. The expanded AR and MA polynomials are the products (1 − Σφᵢ Lⁱ)(1 − ΣΦᵢ L^{is}) and (1 + Σθᵢ Lⁱ)(1 + ΣΘᵢ L^{is}); polynomial multiplication is a join on exponent sums with grouped summation. State dimension is k = max(p + s·P, q + s·Q + 1), unaffected by r. This layer must reproduce, coefficient for coefficient, the `ssm` matrices statsmodels builds for the same spec and parameters — including the `obs_intercept` vector across all t; the fixture generator exports them (§8) and the acceptance test is exact agreement to 1e-14.

### 5.3 Layer 3 — Kalman filter and loglikelihood

Implements the standard covariance Kalman filter as a recursive CTE over t (`USING KEY` on a constant key, carrying the state mean aₜ and covariance Pₜ as lists), with the per-step algebra done either in flattened-list form or by round-tripping through the relational encoding — implementer's choice, but the summation-order rule of §4.2 binds either way. The per-step recursion, written out (predicted-state convention: aₜ, Pₜ are the one-step-ahead predictions entering step t): vₜ = yₜ − dₜ − Z aₜ; Fₜ = Z Pₜ Z′ (H = 0); Kₜ = T Pₜ Z′ Fₜ⁻¹; aₜ₊₁ = T aₜ + Kₜ vₜ; Pₜ₊₁ = T Pₜ T′ − Kₜ Z Pₜ T′ + RQR′ (compute T Pₜ once, reuse; symmetrize Pₜ₊₁ as (P + P′)/2 each step to stop asymmetry drift). The intercept dₜ ≡ 0 when r = 0; the no-exog model is the r = 0 special case of one code path, not a separate implementation. Because dₜ affects only the innovation and not the covariance recursion, Fₜ and the Kalman gain are identical with and without exog at the same ARMA parameters — a property the test suite exploits as a differential check. The loglikelihood accumulates over ALL timesteps t = 1…n with no burn-in: ℓ = −½ Σₜ (log 2π + log Fₜ + vₜ²/Fₜ) — under stationary initialization statsmodels' `loglikelihood_burn` is 0, and the fixture generator asserts this and calls `model.loglike(params, transformed=True)` so probe parameters are interpreted in the constrained space with no transform applied by the reference.

Initialization, v1: the reference configuration pins `simple_differencing=True` in statsmodels, so differencing is applied to the data (Layer 1) and the state is purely stationary ARMA. Initial state mean is zero; initial covariance P₁ solves the discrete Lyapunov equation P = TPT′ + RQR′ via the vec trick, vec(P) = (I_{k²} − T⊗T)⁻¹ vec(RQR′), using Layer 0's Kronecker product and elimination solver. This choice is deliberate: it makes v1's exact-agreement target achievable without diffuse-filter machinery. v2 may add approximate-diffuse initialization (large-κ diagonal) to support `simple_differencing=False`, and exact diffuse thereafter.

The filter must also support a concentrated form in which σ² is profiled out (ℓ maximized analytically over scale given the other parameters); the fixture generator pins `concentrate_scale` explicitly and v1 targets the unconcentrated default, with the concentrated variant behind a config flag as an optimization-speed option that must agree with the unconcentrated optimum at T2 tolerance.

Acceptance: T1 rows of §3 — this layer is where machine-precision agreement lives, and its test suite is the heart of the project.

### 5.4 Layer 4 — Estimation

Starting values are two-stage. First, β⁰ by OLS of the differenced series on the differenced regressors (normal equations via Layer 0), and the regression residuals become the working series. Second, Hannan–Rissanen on those residuals: fit a long AR(m) by OLS with m = max(20, 2·max(p + sP, q + sQ)) truncated to available data, take residuals as ε̂ₜ, then regress on own lags and lagged ε̂ₜ at the multiplicative-seasonal lag positions to obtain initial φ, θ, Φ, Θ, and σ². All regressions use the normal equations assembled as aggregate sums and solved by Layer 0. Initial ARMA values falling outside the stationarity/invertibility region are shrunk toward zero by halving until inside (statsmodels applies a similar guard; exact agreement of starting values is not required and not tested).

The parameter transform replicates statsmodels' stationarity/invertibility enforcement: the Monahan/Jones bijection between unconstrained ℝⁿ and the stationary region via partial autocorrelations (tanh of the unconstrained values as PACFs, then the Durbin–Levinson recursion to polynomial coefficients), applied independently to each of the four polynomial blocks, with the inverse transform for reporting. The β block passes through the transform as identity — statsmodels does not constrain regression coefficients — and σ² is optimized in log space. The transform pair must satisfy round-trip identity to 1e-12 and must match statsmodels' `transform_params`/`untransform_params` on fixture probe points to 1e-10 — this is a T1-class requirement because parameter-space agreement is meaningless without it.

The optimizer is BFGS with a backtracking Armijo line search, run in the unconstrained (transformed) space, entirely as a recursive CTE over iteration number: state is (x, ∇, H⁻¹ approximation) carried as lists; each iteration evaluates ℓ and its gradient by central finite differences with step h = max(1e-7, 1e-7·|xᵢ|) per coordinate. Constants, pinned so two implementations of this spec converge identically in structure: H⁻¹₀ = I; Armijo condition f(x + αp) ≤ f(x) + c₁ α ∇′p with c₁ = 1e-4, initial α = 1, backtracking factor 0.5, at most 30 backtracks per iteration (exhaustion = line-search failure, triggering the restart path); the inverse-Hessian update is the standard BFGS formula H⁻¹ ← (I − ρsy′) H⁻¹ (I − ρys′) + ρss′ with ρ = 1/(y′s), and the update is SKIPPED (H⁻¹ carried unchanged) whenever y′s ≤ 1e-10 · ‖y‖‖s‖, which preserves positive definiteness. The parameter count is r + p + q + P + Q + 1, so exog inflates the gradient cost linearly — each gradient costs 2·(r + p + q + P + Q + 1) filter passes — which the probe-batching requirement absorbs: the batch size grows but remains one query (cross join of probe set × time recursion keyed by probe id via `USING KEY`). β perturbations change only the `obs_intercept` table, so the batched evaluation shares the time-invariant system matrices across all probes that differ only in β; the implementation should exploit this by keying the (T, R, Z, P₁) construction on the ARMA-block hash rather than the full probe id. Convergence: ‖∇‖∞ ≤ 1e-9 or 500 iterations; a restart from a perturbed point (x + uniform(−0.1, 0.1) per coordinate, fixed seed) is attempted once on non-convergence, and the run is flagged. Complex-step differentiation is explicitly rejected for v1 (it would double the arithmetic through the whole filter for precision the T2 target does not require); central differences at the stated step deliver gradient accuracy ~1e-9, sufficient for T2.

Standard errors: numerical Hessian of ℓ in the *untransformed* space by central second differences at θ̂, inverted by Layer 0, negated; delta-method comparison in transformed space is not required.

Acceptance: T2 rows of §3 across the fixture matrix.

### 5.5 Layer 5 — Forecasting, intervals, harness

Multi-step forecasting runs the state recursion forward from the final filtered state with vₜ ≡ 0: aₜ₊ₕ = T aₜ₊ₕ₋₁, ŷₜ₊ₕ = dₜ₊ₕ + Z aₜ₊ₕ, with forecast MSE propagated as Pₜ₊ₕ = T Pₜ₊ₕ₋₁ T′ + RQR′ and Var(ŷ) = Z P Z′. The out-of-sample intercept dₜ₊ₕ = xₜ₊ₕ′β̂ requires the caller to supply future regressor values in the `exog` table through the forecast horizon; the harness validates coverage before forecasting and fails with the missing t range named. Consistent with statsmodels, future exog are treated as known constants — they shift the point forecast but contribute nothing to Var(ŷ), and no parameter-uncertainty term for β̂ is added. Differencing is inverted by h-step cumulative summation anchored on `diff_anchors`, applied D times seasonally and d times ordinarily, with the variance mapped through the corresponding cumulative ψ-weights; the future exog supplied by the caller are on the original scale and are differenced by Layer 1's machinery using the trailing in-sample exog values as anchors, exactly as statsmodels does under `simple_differencing=True`. Because v1 pins `simple_differencing=True`, the statsmodels reference forecasts are on the differenced scale plus its own integration; the fixture generator exports both differenced-scale and original-scale forecasts, and both are tested at T3.

The harness provides: `fit_report` (parameters, SEs, ℓ, AIC, BIC, convergence flag), residual diagnostics, and a grid runner that fits every row of an orders table and ranks by AIC. Information criteria use statsmodels' formulas with the parameter count including σ²: AIC = 2k_params − 2ℓ and BIC = k_params·ln(n_eff) − 2ℓ, where n_eff is the length of the differenced series (the state-space nobs under `simple_differencing=True`) — the fixture-exported values are the arbiter if any counting question arises. Ljung–Box Q is computed on standardized innovations vₜ/√Fₜ at the exact lag set exported in the fixture (the generator records which lags it passed to `acorr_ljungbox`, sidestepping statsmodels' shifting defaults), tolerance rel ≤ 1e-6. The harness also owns input validation with named failures: p, d, q, P, D, Q ≥ 0; s ≥ 2 whenever any of P, D, Q > 0; n after differencing must exceed the AR(m) length needed by Hannan–Rissanen plus the regression degrees of freedom, with the minimum n stated in the error; and the future-exog coverage check of the forecasting path.

## 6. Numerical Considerations

Beyond §4.2's summation ordering: elimination uses partial pivoting always; log Fₜ guards against Fₜ ≤ 0 by failing loudly (a negative innovation variance is a bug, not a condition to smooth over); the Lyapunov solve validates its result by substituting back into P − TPT′ − RQR′ and requiring max |residual| ≤ 1e-9 · max|P|; PACF-transform edge cases near |ρ| = 1 are clamped at tanh's natural range with no additional clipping, matching statsmodels; and all literals are written to parse as DOUBLE (no integer division surprises — use `2.0·pi()` forms). The test suite includes one deliberately near-nonstationary fixture (largest AR root modulus 0.98) and one near-noninvertible fixture (θ = −0.95) to exercise these paths; T2 tolerance is relaxed to 1e-5 for these two fixtures only, T1 is not relaxed.

## 7. Performance Targets

Not a correctness gate, but a design constraint: a single loglikelihood evaluation for n = 500, k = 14 should complete in under 2 seconds; a full fit (HR + ~100 BFGS iterations with batched gradient probes) for the airline-passengers fixture in under 10 minutes on a laptop, measured on DuckDB 1.5.4 at default threading. The performance levers, in priority order: `USING KEY` batching of probe points so one central-difference gradient (2·nparams filter runs) executes as a single recursive query keyed by probe id, with DuckDB parallelizing across probes while each probe's time recursion stays sequential — this is where the 1.5 engine earns its keep, since inter-probe parallelism is safe under §4.2 (ordering is enforced *within* each probe's fold, and probes are independent); the flattened-list state encoding inside the recursion, avoiding per-timestep join planning entirely, with the relational encoding reserved for the one-time Lyapunov solve and elimination where its debuggability matters; the validated fast list kernels of §4.2 where the determinism test admits them; and hash-join-plan assertions on Layer 0 templates (§4.3) so a planner regression to nested loops is caught by CI rather than by a mysteriously slow fit. Each layer's test suite includes an `EXPLAIN ANALYZE`-based timing smoke test with generous thresholds (5× target) to catch order-of-magnitude regressions without flaking on machine variance.

## 8. Validation Fixtures

A Python generator script (the only non-SQL artifact, run once, offline) pins statsmodels and NumPy versions and emits Parquet files per fixture — Parquet rather than CSV so that doubles round-trip losslessly as binary IEEE 754 values with no text-formatting layer between the reference and the comparison; a T1 target of 1e-10 must not be spent on a `%.17g` printing convention. Each fixture contains: the input series and (where applicable) the exog table including future values through h = 36; the model spec and pinned options (`simple_differencing=True`, `enforce_stationarity=True`, `enforce_invertibility=True`, `concentrate_scale=False`, `hamilton_representation=False`, `measurement_error=False`, `mle_regression=True`, `trend='n'`); the ssm system matrices — including the obs_intercept vector for exog fixtures — at 5 probe parameter vectors; ℓ and per-step (vₜ, Fₜ) at 25 probe vectors spanning the interior and near-boundary of the stationary region plus θ̂ (for exog fixtures, probes vary β jointly with the ARMA block); fitted parameters in both spaces, σ̂², ℓ(θ̂), `bse`; forecasts and standard errors to h = 36 on both scales; and Ljung–Box statistics. Fixture matrix, eleven cases: ARMA(1,0,1) sanity on white-noise-plus-AR synthetic data; ARIMA(1,1,1); ARIMA(2,1,2); SARIMA(1,0,0)(1,0,0)₄; SARIMA(0,1,1)(0,1,1)₁₂ on the airline passengers series (the canonical benchmark); SARIMA(1,1,1)(1,1,1)₁₂ synthetic, n = 300; SARIMA(2,1,0)(0,1,1)₁₂; the near-nonstationary case; the near-noninvertible case; ARIMAX(1,1,1) with r = 2 synthetic regressors (one smooth trend-like, one pulse intervention dummy); and SARIMAX(0,1,1)(0,1,1)₁₂ with r = 2 (the airline model plus regressors, exercising exog and seasonal machinery jointly). The SQL test runner reads the Parquet fixtures directly (`read_parquet` with hive-style per-fixture directories), executes each tier, and emits a pass/fail table with observed deltas; it is itself pure SQL plus a shell driver that pins the DuckDB binary version and sets `lambda_syntax = 'DISABLE_SINGLE_ARROW'`.

## 9. Milestones

M0 — Layer 0 complete with its NumPy-fixture test suite green. M1 — Layers 1–2 complete; system matrices, including the obs_intercept vectors of the exog fixtures, match statsmodels exactly on all fixtures. M2 — Filter and loglikelihood pass T1 on the ARMA(1,0,1) and ARIMA(1,1,1) fixtures; this is the project's keystone milestone and the point of maximum debugging. ## 10. Risks and Mitigations

The dominant technical risk is silent numerical divergence in the filter recursion that passes casual inspection but fails T1; mitigation is the per-timestep (vₜ, Fₜ) comparison, which localizes the first diverging step immediately. Second risk: optimizer landing in a different local optimum than statsmodels on multimodal likelihoods (most likely on the seasonal fixtures); mitigation is the ℓ-comparison diagnostic in §3 — if our optimum has ℓ ≥ statsmodels' minus 1e-8, the fixture's parameter row is re-baselined to our optimum with the discrepancy documented, since matching a worse optimum is not a goal. Third risk: recursive-CTE performance making the BFGS loop impractical; mitigation is probe batching and the list encoding, with the fallback of reducing gradient probes via coordinate-wise caching. Exog-specific risk: near-collinear or difference-degenerate regressors making the β estimates ill-conditioned and the Hessian near-singular; mitigation is the Layer 1 rank check with a named-column failure, plus a documented recommendation to standardize regressors (statsmodels comparison then uses identically standardized fixtures, so scaling is never a source of disagreement). Fourth risk: reference or engine version drift — statsmodels changing reference numbers, or DuckDB 2.0 (September 2026) altering semantics the implementation leans on; mitigation is version pinning on both sides (generator lockfile, pinned DuckDB binary in the test driver), committed fixtures that are never regenerated silently, the dual-version CI run of §4.3, and the strict-lambda setting that converts the one known 2.0 removal from a latent failure into an immediate CI error.

## 11. Deliverables

The repository mirrors the duckLM layout — proposed name `duckARIMA` (or `duckTS`), sibling to duckLM, MIT-licensed, "table macros for DuckDB 1.5+, no extensions and no driver required."

`sarimax_macros.sql` — the entire library in one file of `CREATE OR REPLACE MACRO` statements, loaded with `.read sarimax_macros.sql` once per session, callable from the DuckDB CLI or any driver with table and column names passed as strings. Public macros, following duckLM's fit / predict / evaluate / summary verb set: `sarimax_fit(data, y_col, p, d, q, P := 0, D := 0, Q := 0, s := 1, exog_cols := [])` returning the model table; `sarimax_forecast('m', data, y_col, h, newdata := NULL)` for point forecasts with standard errors and intervals on the original scale, `newdata` supplying future exog and required when the model has regressors; `sarimax_summary('m', data, y_col)` for coefficients with SEs, z-statistics, p-values, and confidence intervals, computed with the same in-SQL normal-distribution helpers duckLM established; `sarimax_evaluate('m', data, y_col)` for ℓ, AIC, BIC, residual variance, and Ljung–Box; `sarimax_residuals('m', data, y_col)` for standardized-innovation diagnostics; and `sarimax_grid(data, y_col, orders)` fitting every row of an orders table and ranking by AIC. The model table has a fixed long-form schema — `(kind VARCHAR, name VARCHAR, idx INT, value DOUBLE, value_list DOUBLE[])` — with `kind` ∈ {'param' (β, φ, θ, Φ, Θ, σ² in canonical order, `value` untransformed and one row each), 'param_unc' (the same in unconstrained space), 'spec' (p, d, q, P, D, Q, s, r, n, n_eff), 'meta' (loglik, aic, bic, converged, iterations, grad_norm), 'anchor' (the `diff_anchors` rows, endog and exog, `idx` positional), 'state' (final predicted state aₙ₊₁ and vec(Pₙ₊₁) in `value_list`)} — everything `sarimax_forecast` and the diagnostics macros need with no refit, documented column-by-column in GUIDE.md. Development proceeds in the layered files of §4 (`sql/00_linalg.sql` … `sql/06_harness.sql`) but those are build inputs: a concatenation step assembles the single distributable file, with internal helpers namespaced `_sarimax_*`, and CI verifies the assembled file is what ships.

`tests/` — two independent paths, exactly as in duckLM. A pytest suite (with `tests/requirements.txt` pinning statsmodels and NumPy) that loads the committed Parquet fixtures of §8 and additionally recomputes references live against statsmodels on fixed-seed data, so the frozen-fixture tiers of §3 and a live cross-check both run under `pytest tests/ -q`; the fixture generator lives here too, as the same pinned environment. And `tests/smoke.sql`, a pure-SQL smoke test runnable with `duckdb < tests/smoke.sql` and no Python at all — it fits a small ARIMAX on inline data, forecasts, and asserts sane invariants (convergence flag, finite ℓ, interval ordering), guarding the "no driver required" claim. `tests/README.md` documents both paths and the fixture-regeneration policy (never silently; §10).

`README.md` — duckLM-style: one-paragraph capability summary with the statsmodels-validation claim stated at the honest tolerance tiers of §3 (loglikelihood to machine precision at fixed parameters; fitted parameters and forecasts to optimizer tolerance — not the blanket "machine precision" phrasing), a Setup section that is one `.read` line, a Quick Taste block (`CREATE TABLE m AS SELECT * FROM sarimax_fit('sales', 'units', 0,1,1, P := 0, D := 1, Q := 1, s := 12);` then `sarimax_forecast` and `sarimax_summary`), and a Files section. `CHEATSHEET.md` — one-page signature reference for every public macro. `GUIDE.md` — the user's guide: task-oriented examples, statistical conventions (parameter ordering, differencing anchors, future-exog contract, how intervals treat exog), and the full contract and edge-case behavior including the named-failure conditions of §5.1 and §5.5. `LICENSE` — MIT.

Definition of done: all eleven fixtures green at all tiers under the pytest path, and the smoke test green under the bare CLI path, on a clean DuckDB 1.5.4 session at both `threads=1` and default threading; the same suites executed against the latest current-line release with any failures triaged as upstream-change or own-bug before release; zero deprecation warnings emitted anywhere in a full run; and the README quick-taste block executes verbatim against the shipped `sarimax_macros.sql`.

## 12. Implementation Notes for the Coding Agent

Work in milestone order (§9); do not start a layer before the previous layer's acceptance suite is green, because every downstream debugging session assumes upstream is trusted. The very first commit is the fixture generator plus the committed fixtures — nothing can be validated before the references exist — and the second is `tests/smoke.sql` in failing form, so both test paths run (red) from day one. Fixtures are never regenerated to make a test pass; if a fixture looks wrong, the resolution path is to reproduce the discrepancy in a standalone statsmodels snippet and only then, with the cause documented, regenerate deliberately.

Debugging protocol for T1 failures, which will dominate M2: compare the per-timestep (vₜ, Fₜ) trace against the fixture and find the FIRST diverging t. Divergence at t = 1 means initialization (P₁/Lyapunov, or the system matrices — recheck against the exported ssm first, since §5.2's 1e-14 test localizes that in seconds); divergence that grows from small beginnings means an update-equation error or a summation-order leak; divergence only near the invertibility boundary means the transform or a conditioning issue. Never chase ℓ directly — it is an integral of the trace and hides where the error entered. For T2 failures with T1 green, evaluate our ℓ at statsmodels' θ̂ and statsmodels' ℓ at ours; whichever is higher won, per §10's re-baselining rule.

SQL-specific traps that have bitten before: DuckDB macro parameters are expressions, not identifiers — table-name-as-string arguments require `query_table()` for dynamic table access inside table macros (duckLM's pattern; study `regression_macros.sql` before designing the macro layer, as it is the working precedent for every packaging question here). Recursive CTE column types are fixed by the anchor — cast anchor-term literals to DOUBLE and construct lists with explicit element types, or iteration 2 fails with a type error the anchor hid. `list` aggregation without ORDER BY is nondeterministic exactly like SUM. Integer literals in arithmetic contexts must be written as doubles per §6. And test every layer at both `threads=1` and default before moving on — a threading nondeterminism found at M2 is a one-hour fix, the same bug found at M5 is a week.

When this spec and a fixture disagree, the fixture wins for values and the spec wins for structure; when this spec is silent, match statsmodels' observable behavior and record the decision in GUIDE.md's conventions section rather than leaving it implicit in code.
